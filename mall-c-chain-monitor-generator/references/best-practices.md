# Best Practices

## 1. Always Start From a Business Interface Table

The customer should provide the key business interface table first. Do not batch-create monitors from guesses.

Minimum table information:

- business system name;
- key business chain;
- business suffix for monitor titles;
- key API resources;
- P90/P99 latency thresholds;
- optional HTTP/APM error thresholds.

## 2. Generate One JSON

The standard delivery is one importable monitor JSON containing:

- every key business chain monitor;
- the overall business-system monitor;
- four monitor types for each scope.

For 12 chains, the expected count is:

```text
(12 chains + 1 overall business system) * 4 = 52 monitors
```

## 3. Keep Naming Consistent

Use exactly the same naming style as the reviewed single-chain monitors. Only replace the business suffix.

For overall monitors, the suffix is the business system name, for example `傲雷商城`.

## 4. Use Only One Tag

Every generated monitor should use only the business system name as tag:

```json
[{"name": "傲雷商城"}]
```

Extra tags make customer imports noisier and should not be generated by default.

## 5. Preserve Reviewed Template Style

Use the built-in reviewed 4-checker template unless the user provides a newer exported template.

Preserve:

- `jsonScript.type`;
- message style;
- rule structure and match times;
- `groupBy` dimensions;
- `extend.querylist` layout.

Only replace:

- titles;
- route filters;
- target DQL;
- thresholds;
- sign IDs;
- tagInfo.

## 6. Treat Dynamic Routes Conservatively

Do not put templated routes like `/detail/{id}` directly into `resource IN [...]` only. Convert them to prefix matches:

```text
resource = match('/api/product/api/detail/')
```

This matches real resources such as `/api/product/api/detail/123`.

## 7. Validate DQL Before Delivery

After generating JSON:

1. Export target DQL and querylist DQL with `scripts/export_target_dql.py --include-querylist`.
2. Validate each `.dql` with `dqlcheck`.
3. If a DQL fails, fix the generator or template and regenerate. Do not hand-edit only the final JSON.

## 8. Import Strategy

Before customer import:

- check total monitor count;
- check at least one static-route chain;
- check at least one dynamic-route chain;
- check the overall business-system monitors;
- verify that tagInfo contains only the business system name.

## 9. Iteration Rules

- New chain: update the customer table, regenerate one JSON.
- Threshold change: update table or CLI options, regenerate one JSON.
- Message or rule structure change: update the reviewed template, regenerate one JSON.
- Route matching issue: update resource values or route normalization options, regenerate one JSON.
